# MS365 Tab UI Chat Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make the Microsoft 365 tab chat with the Agent by wiring its composer to the reused Cowork chat machinery through a thin `Ms365ChatController`, tagging MS365 conversations with a new `surface` field so they stay out of the Cowork sidebar, and registering each MS365 session with `Ms365SessionScope`.

**Architecture:** The MS365 tab reuses the Cowork transport (`createSession`/`startEvStream`/`sendSessionMessage`) and persistence (`conversation-controller` + conversation store). A thin `Ms365ChatController` adds only the MS365-specific bits: a connected-gate, `surface:"ms365"` on conversation creation, and one `setMs365SessionScope(sid,true)` call after session creation. A new `surface: "cowork" | "ms365"` field on conversation records (stored in the existing JSON document — no SQL migration) lets the Cowork list filter MS365 conversations out.

**Tech Stack:** TypeScript (NodeNext ESM, `.js` import specifiers), `node:test` via tsx, Electron/loopback service, vanilla-TS renderer.

## Global Constraints

- `setMs365SessionScope` is called ONLY by the MS365 tab; the Cowork flow never calls it, so a Cowork session is never allowed. `Ms365SessionScope` at the router remains the real fail-closed execution guard. (`service/src/ms365/ms365-session-scope.ts`)
- The MS365 composer only enables when `connectionState === "connected"`; unconnected shows the "Chưa kết nối" card (existing behavior). (`app/ui/src/ui-shell/microsoft/ms-assistant-view.ts`)
- `surface` defaults to `"cowork"`; records with the field absent (legacy) read as `"cowork"`. No SQL migration — `surface` rides inside the existing `document_json` shell. (`service/src/db/sqlite-conversation-store.ts`)
- Renderer never touches the DB or secret bytes — only typed preload + capability IPC.
- Provider/model, permission gate, write-mode, `Ms365SessionScope`, the MS365 router/tools/connector, supervisor, and live-launch are all UNCHANGED by this plan.
- ESM `.js` import specifiers; `node:test` (`import { test } from "node:test"`, `import assert from "node:assert/strict"`).
- Commands: `npm run typecheck` (tsc -b), `npm test` (node --test via tsx). Run `scripts\verify-fast.bat` before committing product code. NOTE: `npm test` has ~16 files of KNOWN pre-existing failures (e.g. `streaming-coalesce.test.ts`) unrelated to this work — confirm no NEW failures rather than expecting a fully green suite.

```typescript
/**
 * Thin MS365 chat controller. Reuses the Cowork transport (createSession / stream /
 * sendSessionMessage) and adds only the MS365-specific steps: a connected-gate, a
 * surface:"ms365" conversation, and a one-time Ms365SessionScope allow per session.
 * The router's Ms365SessionScope remains the real execution guard.
 */

import type { ServiceClient } from "./service-client.js";

export interface Ms365StreamHandle {
  stop(): void;
  readonly done: Promise<unknown>;
}

export interface Ms365ChatControllerDeps {
  readonly getClient: () => ServiceClient | null;
  readonly isConnected: () => boolean;
  readonly workspacePath: () => string | null;
  /** Open the live EV stream for a session (inject the real startEvStream at wiring time). */
  readonly startStream: (sessionId: string) => Ms365StreamHandle;
}

export interface Ms365ChatController {
  readonly runtimeSessionId: string | null;
  send(text: string): Promise<void>;
  disconnect(): Promise<void>;
}

export function createMs365ChatController(deps: Ms365ChatControllerDeps): Ms365ChatController {
  let conversationId: string | null = null;
  let sessionId: string | null = null;
  let stream: Ms365StreamHandle | null = null;

  async function ensureSession(client: ServiceClient): Promise<string> {
    if (sessionId !== null) return sessionId;
    const workspace = deps.workspacePath();
    if (workspace === null) throw new Error("Chưa chọn workspace.");
    if (conversationId === null) {
      const conv = await client.createConversation({ workspacePath: workspace, surface: "ms365" });
      conversationId = conv.id;
    }
    const session = await client.createSession({ workspacePath: workspace } as never);
    sessionId = session.sessionId;
    await client.setMs365SessionScope(sessionId, true); // one-time allow
    stream = deps.startStream(sessionId);
    return sessionId;
  }

  return {
    get runtimeSessionId() {
      return sessionId;
    },

    async send(text) {
      if (!deps.isConnected()) return; // connected-gate
      const client = deps.getClient();
      if (client === null) return;
      const sid = await ensureSession(client);
      await client.sendSessionMessage(sid, text);
    },

    async disconnect() {
      const client = deps.getClient();
      if (client !== null && sessionId !== null) {
        await client.setMs365SessionScope(sessionId, false); // revoke
      }
      stream?.stop();
      stream = null;
      sessionId = null;
      conversationId = null;
    },
  };
}
```

> NOTE: `createSession`'s real input type is `CreateSessionInput` and its result is `SessionMeta` — read the actual shape in `service-client.ts` and use the real `session.<id-field>` name (the test assumes `sessionId`; confirm whether it is `sessionId` or `id` and align both the controller and the test to the real field). Same for `createConversation` returning `ConversationRecord` (its id field is `id`). The persistence calls (recordUserMessage / linkRuntimeSession / completeRuntimeTurn) are delegated at WIRING time in Task 6 via the reused conversation-controller — this controller stays thin and only owns the MS365-specific gate + scope + session lifecycle. If the reviewer of Task 6 finds persistence must be threaded through here instead, that is a Task 6 wiring decision, not a change to this unit's contract.
