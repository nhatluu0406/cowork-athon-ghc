# Plan: Convert LanceDB VectorDB from HTTP JSON-RPC to gRPC

## Context

The Go backend communicates with the LanceDB vector service via **JSON-RPC over HTTP** (`LanceDBHTTPClient` in `lancedb_http.go`). The proto contract already exists at `proto/lancedb/service.proto` but is incomplete, no `.pb.go` files are generated, and `google.golang.org/grpc` is not in `go.mod`. The Python server currently serves JSON-RPC over HTTP (via `service.py`) with only a skeleton gRPC server (`server.py`).

**Goal**: Replace the HTTP JSON-RPC client with a native gRPC client. Keep the `vectordb.Client` interface unchanged so callers are unaffected.

## Proto Gaps (already identified)

1. `SearchRequest` in proto missing `repo_id`, `epoch`, `embed_model_id` fields (exist in Go `SearchRequest`)
2. `VectorRecord` in proto missing `product_id` field (exists in Go `VectorRecord`)
3. Proto has `Compact` RPC but Go `Client` interface has `OptimizeIndex` — no mapping
4. Missing `optimize_index` RPC in proto
5. `BatchInsert` needs to be added to Go `Client` interface (it is already there, but proto has it)

## Approach

### Phase 1: Fix proto contract

- File: `src/Backend/proto/lancedb/service.proto`
- Add missing fields to `SearchRequest` and `VectorRecord`
- Add `OptimizeIndex` RPC (maps to Go `OptimizeIndex`)
- Keep `Compact` as-is (optional, not in Go `Client` interface)

### Phase 2: Generate gRPC stubs

- Add `google.golang.org/grpc` and `google.golang.org/protobuf` to `go.mod`
- Install `protoc` + `protoc-gen-go` + `protoc-gen-go-grpc`
- Generate `.pb.go` and `.pb.gw.go` files under `proto/lancedb/`
- Add `buf.gen.yaml` or Makefile target for regeneration (reuse existing Makefile pattern)
- The `go_package` option points to `github.com/dungpd4/rad-system/proto/lancedb`

### Phase 3: Implement gRPC client

- New file: `src/Backend/internal/vectordb/lancedb_grpc.go`
- Implement `vectordb.Client` interface using generated gRPC stubs
- Reuse index-tracking logic from `LanceDBHTTPClient` (`shouldOptimizeIndexLocked`, `indexRebuildThreshold`, `indexRebuildInterval`)
- New constructor: `NewLanceDBGRPCClient(address string, timeout time.Duration) vectordb.Client`

### Phase 4: Wire it up

- Update `NewClient` in `client.go`:
  - Add `"grpc"` backend option alongside `"lancedb"` (HTTP) and `"mock"`
  - `"lancedb"` backend can switch to gRPC by default (configurable)
- Update callers in `cmd/server/main.go:227` and `cmd/standalone/main.go:282` to use gRPC client
- Update `Config.Backend` validation

### Phase 5: Update Python server

- Replace `service.py` (HTTP JSON-RPC) with proper gRPC service using generated protobuf Python stubs
- Wire `server.py` to register the real gRPC service
- Keep `storage.py` unchanged

### Phase 6: Tests & cleanup

- Rewrite `lancedb_http_test.go` as `lancedb_grpc_test.go` (same test coverage)
- Tests that need a real server can use mocking or keep HTTP tests for backward compat
- Remove `lancedb_http.go` once verified

## Questions

1. **Migration strategy**: Should we support both HTTP and gRPC backends simultaneously (with a config flag), or fully replace HTTP with gRPC? This affects whether `lancedb_http.go` is removed or kept as a fallback.
2. **Proto buf toolchain**: Do you want `buf.build` tooling or plain `protoc` for code generation? The project currently has no proto build setup, so either works.
3. **Python gRPC stubs**: Do you need the Python side regenerated too (to serve proper gRPC), or is the current JSON-RPC server adequate for now? If Python also moves to gRPC, we need `protoc --python_out` and `protoc --grpc_python_out`.
4. **TLS/mTLS**: The proto says `add_insecure_port` in server.py. Do you want TLS support for the gRPC channel now, or keep it insecure (internal network only)?
