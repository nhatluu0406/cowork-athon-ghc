# Plan: MCP Merge Assistant — Git-History Chunking + Knowledge Graph

## Context

Build a simplified, standalone system that ingests Git branch history, chunks code at file/symbol level, builds a semantic knowledge graph, and exposes MCP tools for Claude Code to automate merge operations between two branches (`fpt-main-a` → `fpt-main-b`).

**Key differences from RAD Knowledge Gateway:**
- Cloud LLM only (Anthropic for completions)
- C# and C++ only (no Go, Python, TypeScript)
- Merge-focused workflow (no general RAG search)
- MCP as primary interface (no web UI)
- Separate repo — independent project, inspired by RAD patterns
- dotnet + MSBuild for builds
- Embeddings + vector search (cloud embeddings)
- Temp branch by default for merge safety

## Approach

### Phase 1: Project Scaffold & Infrastructure

1. Create new standalone repo structure (independent Go module)
2. Set up Go module with dependencies
3. Configure Anthropic LLM provider (env vars: ANTHROPIC_API_KEY, MODEL)
4. Set up SQLite database with FTS5 + vector storage
5. Write configuration loader (YAML + env vars)

### Phase 2: Git Delta Ingestion

1. Implement `git/client.go` with merge operations:
   - `GetMergeBase()` — `git merge-base A B`
   - `ListCommits()` — `git log merge-base..branch`
   - `GetCommitMeta()` — author, time, message, parents, touched files
   - `GetDiffHunks()` — `git diff OLD..NEW -- path`
   - `GetFileContent()` — `git show COMMIT:PATH`
   - `AttemptMerge()` — `git merge --no-commit --no-ff`
   - `ListConflicts()` — `git diff --name-only --diff-filter=U`
   - Reuse: `git/client.go` pattern from RAD (exec.CommandContext)

2. Implement `ingestor/base_snapshot.go`:
   - Checkout base branch
   - Collect all C#/C++ files
   - Store file inventory in SQLite

3. Implement `ingestor/delta.go`:
   - Fetch commits from incoming branch since merge-base
   - Extract diff hunks per file
   - Store commit metadata + diff hunks in SQLite

### Phase 3: Chunking Engine

1. Implement `chunking/file_chunker.go`:
   - File-level chunking for entire base snapshot
   - Size limits: max 4096 tokens per chunk
   - Overlap: 10 lines between chunks
   - Store chunks in SQLite

2. Implement `chunking/symbol_chunker.go`:
   - Symbol-level refinement for changed/conflict files
   - Reuse C# parser from RAD (`parser/csharp/parser.go`)
   - Reuse C++ parser from RAD (`parser/cpp/parser.go`)
   - Extract methods, classes, interfaces, structs, enums
   - Handle oversized symbols (flag + summary)

3. Implement `chunking/provenance.go`:
   - Link chunks → commits (ProvenanceLink)
   - Link chunks → diff hunks

### Phase 4: Semantic Graph Builder

1. Implement `graph/builder.go`:
   - Build graph nodes from symbols
   - Edge types: calls, depends_on, extends, implements, inherits
   - Reuse RAD graph types (`graph/types.go`)
   - Reuse edge resolvers pattern (`graph/resolvers/`)

2. Implement `graph/csharp_edges.go`:
   - C# specific: method calls, interface implementations, inheritance
   - Regex-based for speed (LSP optional later)

3. Implement `graph/cpp_edges.go`:
   - C++ specific: template instantiations, includes, inheritance
   - Handle header/source file pairing

4. Implement `graph/query.go`:
   - Impact analysis: traverse from conflict node downstream
   - Find callers of changed symbols
   - Find tests that reference changed symbols

### Phase 5: Conflict Detection & Analysis

1. Implement `conflict/detector.go`:
   - Run `git merge --no-commit` in controlled workspace
   - Parse conflict markers from files
   - Build `MergeConflictRecord` per conflict

2. Implement `conflict/classifier.go`:
   - Classify conflicts: trivial (overlap), structural (same symbol), semantic (same logic)
   - Score ambiguity (0-1): overlap=0.1, structural=0.5, semantic=0.9
   - Flag high-ambiguity for human review

3. Implement `conflict/context.go`:
   - Given conflict ID, retrieve:
     - File/symbol context (chunks)
     - Related commits from incoming branch
     - Impact analysis (downstream callers)

### Phase 6: MCP Tools Implementation

Register tools in `mcp/tools.go` following RAD pattern (`mcp/handlers/tools.go`):

**Analysis Tools:**
- `merge.get_context(base, incoming)` — merge-base + overview
- `merge.get_merge_base(base, incoming)` — compute merge base SHA
- `merge.list_changed_files(base, incoming)` — diff file list
- `merge.get_diff_hunks(path)` — hunks for a file
- `kb.get_file_chunk(path)` — file-level chunk
- `kb.get_symbol_chunks(path, symbol?)` — symbol chunks
- `kb.search_symbols(query)` — semantic search
- `merge.list_incoming_commits()` — commits since merge-base
- `merge.analyze_conflicts()` — conflict records
- `merge.get_conflict_context(conflict_id)` — full context
- `merge.impact_report(path|symbol)` — downstream impact

**Editing Tools:**
- `workspace.apply_patch(patch)` — apply unified diff
- `workspace.edit_file(path, edits)` — structured edits
- `workspace.read_file(path)` — read file content
- `workspace.find_conflict_markers()` — list markers

**Quality Tools:**
- `quality.run(gates, scope)` — run quality gates
- `quality.get_report(run_id)` — retrieve report

**Git Tools:**
- `git.status()` — current status
- `git.diff(staged)` — working tree diff
- `git.add(paths)` — stage files
- `git.commit(message)` — create commit
- `git.checkout(ref)` — switch branch
- `git.merge(ref, no_commit)` — attempt merge
- `git.abort_merge()` — abort merge

**Guardrails:**
- `workspace.assert_clean()` — verify clean state
- `merge.begin_session()` — start merge session (lock)
- `merge.end_session()` — end session, cleanup

### Phase 7: Quality Gate Runner

1. Implement `quality/gates.go`:
   - Build gate: `dotnet build` (C#) + `msbuild` (C++)
   - Lint gate: read repo config for tools
   - Test gate: `dotnet test` + MSBuild test runner
   - Complexity gate: basic cyclomatic complexity per file

2. Implement `quality/reporter.go`:
   - Normalize output to `QualityReport` struct
   - Compare against baseline
   - Flag new failures

### Phase 8: Merge Orchestrator

1. Implement `orchestrator/session.go`:
   - State machine per spec (INIT → DONE/FAILED)
   - Workspace isolation (temp branch)
   - Lock management (file lock)

2. Implement `orchestrator/commit_planner.go`:
   - Plan PR-style commit series:
     1. Mechanical changes (safe refactors)
     2. Compile fixes
     3. Test fixes
     4. Cleanup/refactor
     5. Final gates pass

### Phase 9: Cloud LLM Integration

1. Implement `llm/anthropic.go`:
   - Anthropic Messages API for completions (default model: Claude)
   - Streaming support for long responses
   - Tool use support (MCP-native pattern)

2. Implement `llm/embedder.go`:
   - Use Claude to generate embeddings: prompt Claude to output a fixed-dimensional numeric vector for each text chunk
   - Store vectors in SQLite (FTS5 + vector cosine similarity)
   - Note: Claude embeddings are more expensive per-token than dedicated embedding APIs, but avoids extra API keys

3. Implement `llm/config.go`:
   - Env var: `ANTHROPIC_API_KEY`, `LLM_MODEL` (default: claude-sonnet-4-20250514)
   - Embedding model config (can use different Claude model for embeddings if cheaper)
   - Timeout, retry, rate-limit settings

### Phase 10: Database Schema

SQLite tables:
- `repos` — repository metadata
- `branches` — branch references (name, head_sha)
- `commits` — commit metadata
- `files` — file inventory
- `chunks` — file/symbol chunks
- `symbols` — extracted symbols
- `graph_nodes` — graph nodes
- `graph_edges` — graph edges
- `provenance` — chunk→commit links
- `conflicts` — merge conflict records
- `quality_reports` — gate results
- `sessions` — merge session state
