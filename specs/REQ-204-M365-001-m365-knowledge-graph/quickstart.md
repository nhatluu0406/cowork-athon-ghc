# Quickstart: Enterprise Knowledge Graph from Microsoft 365

**Phase**: 1 | **Audience**: frontend integrators, anyone standing up the backend locally for the first time

## 1. Prerequisites

- Go 1.22+
- Rust 1.75+ (for `llm-svc` — see spec.md §3.5)
- PostgreSQL 14+ (locked decision — not SQLite; see memory `req204-mergeassistant-critical-constraints` and spec.md provenance note)
- Neo4j 5.x
- `llm-svc` running locally (or reachable over gRPC) — the Go backend no longer talks to any LLM provider directly (spec.md §3.3/§3.5, amendment 2026-07-11)
- (Optional for local dev) A Microsoft Entra ID app registration — otherwise use `M365_AUTH_MODE=jwt_demo`

## 2. Environment setup

**Go backend** (`src/m365-knowledge-graph/.env`) — copy from plan.md / spec.md §12. Note: as of the gRPC amendment, Go holds **no** LLM provider credentials — those moved to `llm-svc` (§12.1):

```bash
HOST=0.0.0.0
PORT=8080
DATABASE_URL=postgres://user:pass@localhost:5432/m365kg
M365_TENANT_ID=<tenant>
M365_CLIENT_ID=<client-id>
M365_CLIENT_SECRET=<secret>
M365_AUTH_MODE=entra_id   # or jwt_demo for local dev without a tenant
NEO4J_URI=bolt://localhost:7687
NEO4J_USERNAME=neo4j
NEO4J_PASSWORD=<password>
LLMSVC_ADDR=localhost:9090
LLMSVC_TLS=false
JWT_SECRET=<auto-generated-if-omitted>
ALLOWED_ORIGINS=http://localhost:5173
DELTA_SYNC_INTERVAL=5m
```

**`llm-svc`** (`llm-svc/.env`) — copy from spec.md §12.1:

```bash
NLP_MODE=2   # 1=cloud_only, 2=cloud_with_local_preprocess (default), 3=local_only
LLM_API_BASE_URL=https://mkp-api.fptcloud.com/v1
LLM_API_KEY=<api-key>
LLM_MODEL=gpt-4o-mini
LLM_EMBED_MODEL=text-embedding-3-small
BRAIN_LOCAL_PROVIDER=qwen3-8b-q4
BRAIN_FALLBACK_TO_CLOUD=true
```

**Never hard-code these** — CLAUDE.md §8 flags hard-coded hosts/ports as a common mistake to avoid.

## 3. First run (backend)

```bash
cd src/m365-knowledge-graph
go mod tidy
go run ./cmd/server            # runs migrations from internal/metadata/schema.go on startup
```

Verify: `curl http://localhost:8080/api/stats/overview` should return zeroed counts (no data ingested yet).

## 4. Connect an M365 source

```bash
curl -X POST http://localhost:8080/api/m365/connect \
  -H "Authorization: Bearer <token>" \
  -d '{"name":"Engineering OneDrive","type":"onedrive","tenant_id":"<tenant>","config":{"site_id":"<site>"}}'
```

Then trigger a sync:

```bash
curl -X POST http://localhost:8080/api/m365/sync -H "Authorization: Bearer <token>"
```

Watch progress over WebSocket (`ws://localhost:8080/ws?token=<token>`, event `sync_progress`) — see contracts/api.md.

## 5. Ask a question (once ingestion + extraction has run at least once)

```bash
curl -X POST http://localhost:8080/api/knowledge/query \
  -H "Authorization: Bearer <token>" \
  -d '{"query":"Who is working on the Phoenix project?"}'
```

Expected response shape per contracts/api.md `POST /api/knowledge/query`: `answer`, `sources[]`, `entities[]`, `intent`, `latency_ms`. p95 latency target is ≤30s (plan.md Performance Goals).

## 6. Frontend integration

The frontend (`src/Frontend/` or a separate repo per plan.md) talks to the backend via:
- `VITE_API_BASE_URL` / `VITE_WS_URL` env vars (CLAUDE.md §7 — never hard-code `localhost:8080`)
- TanStack Query hooks for all server state (`useKnowledgeQuery`, `useEntities`, `useFeedback`) — never Zustand for server data (CLAUDE.md §6)
- The shared `useWebSocket` hook for real-time sync/query progress — never instantiate `new WebSocket(url)` directly

Minimal integration test scenario (maps to spec.md §16 E2E acceptance flow, steps 1-7):
1. Log in (`/api/auth/login`) → store JWT.
2. Configure a connection (`/api/m365/connect`).
3. Trigger sync (`/api/m365/sync`), observe `sync_progress` WebSocket events.
4. Poll `/api/entities?type=document` until non-empty.
5. Poll `/api/entities?type=person` / `?type=project` until non-empty (confirms NLP extraction ran).
6. Call `/api/graph/nodes` and `/api/graph/edges` — confirm graph populated.
7. Call `/api/knowledge/query` with a question about ingested content — confirm a cited answer comes back.

## 7. Independence check (MergeAssistant)

Before committing any change under `src/m365-knowledge-graph/`, confirm no accidental coupling was introduced:

```bash
grep -r "MergeAssistant" src/m365-knowledge-graph/ || echo "clean"
grep -r "src/Backend\|src/Frontend" src/MergeAssistant/ || echo "clean"
```

Both must print `clean` — see memory `req204-mergeassistant-critical-constraints`.
